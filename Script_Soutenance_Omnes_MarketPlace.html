Script Oral — Soutenance Omnes MarketPlace

Come, David, Hippolyte — Mars 2026

1. Introduction

Bonjour, nous sommes Come, David et Hippolyte, et nous vous presentons Omnes MarketPlace. C'est une plateforme de marketplace etudiante pour la communaute Omnes Education, dans l'esprit du Bon Coin. Les etudiants peuvent acheter et vendre tout type d'article : electronique, vetements, livres, mobilier, equipements sportifs, et plus encore.

Le projet a ete developpe en stack 100% native, sans aucun framework back-end ni CMS. On utilise PHP 8 pour le serveur, MySQL 8 pour la base de donnees, et Bootstrap 5.3 avec du JavaScript et jQuery pour le front. Le tout represente 39 fichiers PHP, 14 tables en base, environ 2 600 lignes de CSS et 800 lignes de JavaScript.

L'application gere trois roles utilisateurs : administrateur, vendeur et acheteur. Et trois types de vente : l'achat immediat, la negociation entre vendeur et acheteur, et le systeme de meilleure offre par encheres a l'aveugle. On va vous detailler tout ca dans la presentation.

2. Repartition du travail

Pour la repartition, Come s'est occupe de l'architecture generale, la base de donnees, la configuration serveur et l'integration globale des differents modules. David a pris en charge tout le front-end : le design CSS, le responsive, la navbar, le footer, les pages publiques et l'identite visuelle du site. Hippolyte a developpe la logique metier : le systeme d'authentification, la gestion des roles, les tableaux de bord admin et vendeur, ainsi que le systeme de paiement et les notifications. On a tous collabore sur les parties transversales comme le panier, les negociations et les tests.

3. Architecture du systeme

L'architecture suit un modele MVC simplifie en PHP natif. On a trois couches : la couche presentation avec HTML, CSS, Bootstrap et JavaScript ; la couche logique avec des scripts PHP d'action qui traitent les formulaires et redirigent ; et la couche donnees avec MySQL via PDO.

Cote structure de fichiers, on a un dossier config pour la connexion PDO, un dossier includes avec les composants partages header, navbar et footer que chaque page inclut. Le dossier pages contient 12 pages publiques, 5 pages admin et 6 pages vendeur. Le dossier php contient 11 scripts d'action. Et on a un fichier CSS unique et un fichier JS unique pour tout le site. Chaque page definit une variable base_url selon sa profondeur dans l'arborescence, et les formulaires font un POST vers les scripts d'action qui traitent la requete puis redirigent.

4. Modele Entite-Association

La base de donnees comporte 14 tables. Au coeur, on a la table utilisateurs qui gere les trois roles avec un hashage bcrypt des mots de passe. La table articles stocke les annonces avec leur type de vente, leur gamme et leur statut. On a aussi une table article_images pour la galerie multi-photos.

Pour les transactions, le point le plus complexe c'est le panier polymorphe. Une seule table panier gere les trois types de transaction grace a deux colonnes nullable : negociation_id et enchere_id. Si les deux sont null, c'est un achat immediat. Si negociation_id est rempli, c'est une negociation validee avec le prix accorde. Si enchere_id est rempli, c'est une enchere remportee. Ca evite d'avoir trois tables panier separees.

On a aussi les tables negociations et negociation_messages pour le chat de negociation avec un maximum de 5 tours. La table encheres stocke les offres a l'aveugle avec le montant maximum et le prix paye. Les tables commandes et commande_articles gerent les commandes finalisees. Et enfin les tables avis pour les notations, notifications pour les alertes, cartes_bancaires pour la simulation de paiement, et cartes_reduction pour les bonus de fidelite.

6. Specifications fonctionnelles

Commencons par les trois types de vente. Le premier, l'achat immediat, c'est le plus simple. L'acheteur voit un prix fixe, clique sur ajouter au panier, et procede au paiement. L'article passe au statut vendu a la finalisation. On verifie qu'il est encore disponible avant chaque ajout.

Le deuxieme type, la negociation, c'est la fonctionnalite la plus complexe du projet. L'acheteur ouvre une session de negociation avec le vendeur. Ils echangent des propositions de prix dans un chat, avec un maximum de 5 tours. Si le vendeur accepte une proposition, l'article est verrouille au prix accorde et ajoute automatiquement au panier de l'acheteur. Si l'acheteur retire ensuite l'article du panier, la negociation expire et l'article est remis en vente avec une notification au vendeur. Tout ca est gere par des transactions SQL avec SELECT FOR UPDATE pour eviter les conflits de concurrence.

Le troisieme type, la meilleure offre, utilise un mecanisme d'enchere a l'aveugle inspire du modele Vickrey. Chaque acheteur soumet son montant maximum sans voir les autres offres. A la cloture par l'administrateur, le gagnant est celui avec l'offre la plus haute, mais il paie le prix de la deuxieme meilleure offre plus un euro, ou le prix de reserve s'il est seul. L'article est ensuite ajoute automatiquement au panier du gagnant, et les perdants recoivent une notification.

Pour le paiement, c'est une simulation complete. L'utilisateur remplit un formulaire d'adresse et de carte bancaire. On valide le format en front et en back : 16 chiffres pour le numero, 3 ou 4 pour le CVV, date d'expiration non depassee, code postal 5 chiffres. Le tout se fait dans une transaction SQL atomique : creation de la commande, insertion des lignes, passage des articles en vendu, et vidage du panier. Si le montant depasse 100 euros, une carte de reduction de 10% est generee automatiquement, et 20% au-dela de 200 euros. Un email de confirmation est envoye via MailHog.

On a aussi un systeme de notifications stockees en base avec un polling AJAX qui met a jour le compteur dans la navbar. Un systeme d'avis avec des notes de 1 a 5 etoiles, un seul avis par utilisateur par article. Une recherche avec filtres avances par categorie, gamme, type de vente et fourchette de prix. Et un profil vendeur personnalisable avec upload de photo et image de fond, avec validation MIME et controle de taille.

8. Securite et bonnes pratiques

Cote securite, toutes les requetes SQL utilisent des requetes preparees PDO avec des parametres nommes, ce qui bloque les injections SQL. Les mots de passe sont hashes avec bcrypt via password_hash. On regenere l'identifiant de session apres chaque connexion pour empecher la fixation de session. Les uploads sont valides par type MIME, extension et taille. Chaque page protegee verifie le role en session. Les messages d'erreur sont generiques cote client et les details sont logues cote serveur. Et les operations critiques utilisent des transactions SQL avec SELECT FOR UPDATE pour gerer la concurrence.

Pour les bonnes pratiques, on a centralise le code commun dans un fichier helpers.php avec des constantes et des fonctions partagees pour eviter la duplication. On utilise des variables CSS pour la coherence visuelle. Les pages sont separees des scripts d'action pour garder une separation claire entre affichage et logique.

11. Bilan

Pour le bilan collectif, le projet a ete formateur. On a du concevoir une application complete de zero sans framework, ce qui nous a force a comprendre en profondeur chaque composant. Les points forts : l'application est fonctionnelle avec les trois types de vente operationnels, l'interface est soignee et responsive, et la securite a ete prise en compte. Les difficultes principales ont ete la gestion de la concurrence sur les negociations et encheres, et le panier polymorphe qui gere trois types de transaction dans une seule table.

Comme ameliorations possibles, on pourrait ajouter des WebSockets pour le temps reel au lieu du polling AJAX, integrer un vrai systeme de paiement comme Stripe, ajouter des tokens CSRF sur tous les formulaires, et mettre en place des tests automatises avec PHPUnit.
